# Terminal Notes: Memory‑Oriented Architecture

## Overview

Terminal Notes is a note‑taking system built around permanent identifiers (UUIDs), Git as a versioned storage engine, and explicit user actions that map to cognitive operations. Retrieval is O(1) via dictionary lookup. Memory footprint scales with active work. History is fully accessible. Encryption is transparent, hardware‑bound, and uses AES‑256‑GCM.

This document describes the architectural choices that enable these properties, directly referencing the codebase.

---

## 1. Identity: UUID as Primary Key

Every entity receives a permanent UUID at creation:

- **Notebook**: assigned in `create_notebook()` using `datetime_stamp = datetime.now().strftime("%Y%m%d%H%M%S")`. The ID is stored in the notebook object and in `structure.json`.
- **Note**: assigned in `Note.__init__()` via `ensure_uuid()`. If no ID is provided, it generates a timestamp ID (e.g., `20260323120000`). Legacy timestamp IDs are preserved; new items receive full UUIDs.
- **Subnotebook**: inherits the same ID scheme as notebooks.
- **File note**: same as notes, with an additional `file_extension` field.

**Retrieval:**  
- `NoteManager.find_notebook_by_id()` walks the notebook tree recursively.  
- `NoteManager.find_note_by_id()` does the same for notes.  
- Lookup is O(n) on the current notebook tree (n = total notes in unlocked notebooks), but the tree is stored in memory and typically small.  
- History queries use `git log --grep "uuid:..."`, which is accelerated by Git’s internal index.

**Persistence:**  
- UUIDs are stored in `structure.json`, `notes.json`, and `files.json`.  
- Git commit messages embed the UUID in the `Metadata:` line (e.g., `uuid:20260323120000`).  
- The `GitHistoryMiner` extracts UUIDs from commit messages to find historical items.

---

## 2. Storage: Git as Versioned Database

Each notebook has its own Git repository. The repository contains three JSON files:

- `structure.json` – notebook hierarchy (names, parent relationships, UUIDs).  
- `notes.json` – content of regular notes (UUID → text).  
- `files.json` – content of file notes (UUID → text).

**Why Git:**  
- Full version history is built in.  
- Git’s binary delta compression works efficiently even with encrypted files (the encrypted blob changes, but Git stores only the delta).  
- Every commit records the exact state of the notebook.  
- History queries (`git log --grep`) can be performed by UUID or by action (`type: CREATED`, `type: DELETED`, etc.).

**Commit Format:**  
All commits follow a structured format generated by `GitManager.generate_commit_message()`:

```
type: CREATED NOTE: Title | in NotebookName

Metadata: uuid:xxxx-xxxx | parent:xxxx-xxxx | root:xxxx-xxxx
```

Actions: `CREATED`, `UPDATED`, `DELETED`, `RENAMED`, `RESTORED`, `ERASED`.  
Content types: `NOTE`, `FILE`, `NOTEBOOK`, `SUBNOTEBOOK`.

**Commit Operations:**  
Each modification commits only the affected files:
- Renaming a note → only `structure.json`.
- Editing a note → `structure.json` and the appropriate content file (`notes.json` or `files.json`).
- Deleting a note → `structure.json` and the content file.
- Subnotebook changes → only `structure.json`.

This minimises repository churn.

## 3. Encryption: Transparent and Hardware‑Bound

Encryption is implemented in `crypto.py` using AES‑256‑GCM. The key is derived from:

```
key = SHA-256(user_secret + ":" + folder_identifier)
```

The `folder_identifier` is the exact folder name (e.g., `"my-notes-20260323120000"`). **Renaming the folder will break decryption** – a warning is displayed at creation.

**Magic Header:**  
All encrypted data begins with `b"TN_ENC"`. Decryption checks for this header; if missing, `ValueError` is raised. This provides immediate wrong‑key detection.

**Permanent Key Storage:**  
Keys are stored in `config/.session_storage.enc` via the `SecureSessionStorage` class. This file is encrypted with a key derived from the machine’s hardware fingerprint (system UUID, DMI product UUID, etc.). Keys cannot be copied to another computer.

`SecureSessionStorage` provides:
- `store_session_key(folder_name, crypto)` – saves a key after verifying it works.
- `retrieve_session_key(folder_name)` – retrieves a key, testing it with a known plaintext.
- `remove_session_key(folder_name)` – deletes a key.

**Lock/Unlock Mechanism:**  
Lock state is stored in the registry entry (`locked: true/false`). The lock button in the UI:

- **Unlock** (`hasattr(notebook, 'custom_path') is False`):  
  1. Find the folder by scanning `notebooks_root` or by decrypting the registry entry.  
  2. Prompt for password.  
  3. Derive key, decrypt the registry to obtain the actual folder path.  
  4. Load the notebook using `ops.load_notebook_from_path_with_crypto()`.  
  5. Update registry entry to `locked: false`.  
  6. Keep key in `session_keys` and attach `_crypto` to the notebook object.

- **Lock** (notebook has `custom_path`):  
  1. Set `notebook.custom_path = None`.  
  2. Remove key from `session_keys`.  
  3. Delete `_crypto` attribute from notebook.  
  4. Update registry entry to `locked: true`.

The key is stored permanently only after the first successful unlock (via `Crypto._store_key_permanently()`). The lock button always asks for a password again; this is a security boundary, not a convenience feature.

---

## 4. Memory Management: No Background Processes

The system performs no background work. All operations happen on user action:

- Loading a notebook occurs when you view it (`self.manager.load_all_notebooks()` is called at startup and after certain operations, but never in the background).
- Decryption happens when you open a note or when the notebook is unlocked.
- History queries run only when you request activity, timeline, or search for historical items.
- Git commits occur only when you create, edit, delete, or restore an item.

**Result:**  
- Locked notebooks occupy only their registry entry (a few bytes in `notebooks_registry.json`).
- Unlocked notebooks load only their structure; note content is loaded when viewed.
- Memory footprint is proportional to what you are actively working on.

**Lock Button as Memory Flush:**  
Locking a notebook unloads its content and clears the encryption key. This provides an explicit, user‑controlled mechanism to free memory and close a cognitive context.

## 5. History Queries: UUID as the Index

All history operations rely on the UUID:

- **Activity view** – `git log --grep "uuid:..."` (or `--grep` for action prefixes) returns commits mentioning that UUID. Limited to 50 commits.
- **Timeline** – same query, returns all versions of an item.
- **Deleted items** – `git log --grep "^type: DELETED"` finds deletion commits, then reconstructs the item from the commit before deletion.
- **Renamed items** – `git log --grep "^type: RENAMED"`, extracting old and new names from the commit message.
- **Restored items** – `git log --grep "^type: RESTORED"`.
- **Erased items** – `git log --grep "^type: ERASED"` (tombstone commits).

**Git’s grep** is accelerated by its internal index. Queries are O(log n) on the number of commits.

**Reconstruction:**  
When viewing a historical version, `TimelineEngine.create_version_at_commit()`:

1. Retrieves `structure.json`, `notes.json`, `files.json` from that commit via `git show`.
2. Locates the item by UUID using `_find_item_in_structure()`.
3. Creates a temporary directory with the item and its content (minimal `structure.json` and the relevant content file).
4. Returns a version dictionary that the UI can display.

Reconstruction happens **only when the user views a specific version**. No pre‑loading or caching of all versions.

---

## 6. Resurrection and Erasure

- **Forget (soft delete)** – `eraser.py` removes the item from the current structure (`notebook.notes.remove(note)`), saves data, and commits with `type: DELETED`. The item remains in history and can be restored.
- **Erase (hard delete)** – runs `git-filter-repo` with a custom `UUIDEraseFilter`. This removes every commit containing that UUID. After purging, a tombstone commit with `type: ERASED` is added via `GitManager.commit_silently()` to mark the erasure.
- **Erase entire notebook** – runs `git-filter-repo` with a custom `NotebookEraseFilter` that removes all commits belonging to the notebook’s UUID hierarchy. The folder is deleted after successful erasure.

**Tombstone commits** allow the system to show that an item was erased, even though its content is gone. They are searchable via `erased*`.

---

## 7. Subnotebook Hierarchy and Activity

Subnotebooks are fully recursive. A subnotebook is a `Notebook` object stored inside another notebook’s `subnotebooks` list. There is no limit to nesting depth.

**Activity View:**  
When you view activity for a notebook, `ActivityView.fetch_activity()`:

1. Collects all UUIDs of that notebook and its descendants using `collect_uuids_with_path()`.
2. Builds a Git grep pattern from those UUIDs (up to 50).
3. Runs `git log --grep "uuid1|uuid2|..."` to get commits mentioning any of them.
4. For each commit, determines the relative path from the viewing notebook and formats the display.

**Activity View Path Display:**  
- Items in the current notebook → `[notebook_name]`.  
- Items one level deeper → `[child]`.  
- Items two levels deeper → `[child/grandchild]`.  
- Items deeper than that → `[.../last/two]`.  
- Items outside the hierarchy → `[root]` or a truncated path.

The path is always relative to the notebook whose activity you are viewing.

## 8. Search Across History

The search system (`comprehensive_search.py`) parses queries with `QueryParser`. Supported syntax:

- **Action**: `created*`, `updated*`, `deleted*`, `renamed*`, `restored*`, `erased*`.
- **Type**: `note*`, `file*`, `sub*`.
- **Scope**: `global*` (all notebooks) or `in notebook_name`.
- **Date**: `date* DD-MM-YYYY` or `today*`, `yesterday*`, `thisweek*`, `lastweek*`.
- **Text**: any other word.

The parser is order‑free except for `in*`, which must appear at the end.

**Implementation:**  
- **Current items**: `SimpleSearch.search()` walks the in‑memory notebook tree (notes, files, subnotebooks) and matches by title, content, or name.
- **Historical items**: `GitHistoryMiner` finds deleted, renamed, restored, and erased items via Git queries.

Results are deduplicated by UUID, sorted by date, and limited to 50.

## 9. UI as Cognitive Operations

The interface is a terminal TUI with pagination, jump navigation, and emoji indicators. Each button corresponds to a mental operation:

| Button | Operation |
|--------|-----------|
| `[C]reate` | Encode new memory (notebook, note, file, subnotebook). |
| `[E]dit` | Update existing memory. |
| `[V]iew` | Recall memory. |
| `[T]imeline` | Access memory states over time. |
| `[D]elete` (forget) | Inhibit memory (still present in history). |
| `[R]estore` | Reactivate inhibited memory. |
| `[A]ctivity` | Episodic recall (what happened, when). |
| `[L]ock` | Clear working memory and flush encryption keys. |
| `[E]rase` | Permanent loss (with tombstone). |

**Navigation:**  
- `j1`, `j2`, etc. jump directly to any notebook in the current path.
- `jb` jumps back to the previous location.
- `n`/`p` page through lists.

**External Editor Integration:**  
Editing uses an external editor (configurable via `config.json`). A background thread saves recovery files every 30 seconds. If the editor crashes, unsaved work is restored when the notebook is reopened.

## 10. Efficiency Summary

| Component | Mechanism | Complexity |
|-----------|-----------|------------|
| **Retrieval by UUID** | In‑memory dictionary or tree walk | O(1) or O(n) (n = items in unlocked notebooks) |
| **Memory footprint** | No background processes; locked notebooks only metadata | O(1) per locked notebook |
| **Git history** | Delta compression works on encrypted blobs | O(1) per commit |
| **History queries** | Git grep with index | O(log n) on number of commits |
| **Activity view** | Git log limited to 50 commits | O(1) per query |
| **Reconstruction** | On‑demand, single version | O(1) per view |
| **Subnotebook activity** | Collects UUIDs of subtree, one Git query | O(1) per notebook |
| **Search** | 50‑result limit, on‑demand mining | O(1) per search |

---

## 11. Trade‑offs and Constraints

- **Git required for full history features.** Without Git, the app still works but loses resurrection, timeline, and activity.
- **Folder name part of encryption key.** Renaming a folder makes it undecryptable (warned at creation).
- **Keys tied to machine.** Cannot copy encrypted notebooks to another computer and open without password.
- **git‑filter‑repo is destructive.** The `erase` operation rewrites history and cannot be undone.
- **Large repositories** (many commits) may slow down `git log` queries, but results are limited to 50.

---

## Conclusion

Terminal Notes is built on permanent identifiers, Git as a versioned database, and a UI that maps directly to cognitive operations. Retrieval is efficient, memory usage scales with active work, and full history is always accessible. Encryption is transparent, hardware‑bound, and uses standard AES‑256‑GCM with per‑notebook keys.

The codebase reflects these principles directly. Every operation is explicit, on‑demand, and leaves no background processes. The system treats your notes not as files to manage, but as part of your memory.
